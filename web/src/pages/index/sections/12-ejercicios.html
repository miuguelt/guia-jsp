De Monolito (JSP) a Microservicios (FastAPI + React)

Hasta ahora has construido un monolito JSP: todo en un solo WAR (lógica, vistas, auth con sesiones). Ahora evolucionamos a una arquitectura de microservicios donde cada componente es independiente, escalable y desplegable por separado. La base de datos y las reglas de negocio se mantienen — lo que cambia es cómo se comunican las piezas.

Comparación Detallada: Monolito vs Microservicios

Dimensión 🏢 Monolito JSP 🔗 Microservicios FastAPI + React
Empaquetado 1 solo archivo WAR desplegado en Tomcat N contenedores Docker independientes
Comunicación Llamadas directas entre clases (Java methods) HTTP/REST entre servicios (JSON sobre la red)
Estado de sesión Stateful: HttpSession en memoria del servidor Stateless: JWT en localStorage del cliente
Frontend Server-side: JSP + JSTL renderiza HTML en Tomcat Client-side: React renderiza componentes en el navegador
API No hay API explícita (Servlet + JSP directo) API REST documentada con Swagger/OpenAPI
Base de datos Compartida: todos los DAOs apuntan a la misma BD Por servicio: cada microservicio puede tener su propia BD
Escalabilidad Vertical: escalas todo el WAR o nada Horizontal: escalas solo el servicio con demanda
Despliegue Reemplazar WAR completo → reiniciar Tomcat Actualizar solo el contenedor del servicio afectado
Tecnología Java obligatorio para todo el stack Cada servicio elige su tecnología óptima
Testing Manual (abrir navegador, probar CRUD) Automatizado: pytest para API, Vitest para React
Equipo Equipo full-stack Java (perfil único) Equipos especializados: backend, frontend, DevOps
Tiempo de inicio 30-60s (iniciar Tomcat + cargar WAR) 2-5s por servicio (contenedor ligero)

Estrategia de Migración: Strangler Fig Pattern

No migres todo de golpe. Usa el Patron Strangler Fig (Higuera Estranguladora): reemplaza el monolito pieza por pieza mientras ambas versiones coexisten.

1
Identificar

Encuentra el módulo con menos dependencias (ej. Catálogo de productos)

2
Extraer

Crea el microservicio con su propia API y BD. Ambos sistemas apuntan a la misma BD inicialmente

3
Redirigir

El frontend (React) consume el nuevo microservicio. El monolito deja de recibir tráfico para ese módulo

4
Remover

Elimina el código antiguo del monolito. Repite con el siguiente módulo hasta que el monolito desaparezca

Lo que SE MANTIENE igual
  • PostgreSQL — misma BD, mismas tablas, mismos datos
  • Reglas de negocio — validaciones, cálculos, lógica de inventario
  • Modelo de datos — roles, usuarios, productos (PK/FK iguales)
  • JWT Secret Key — misma clave firmada, mismo token portable
  • Frontend React — puede consumir TANTO el JSP como FastAPI
Lo que CAMBIA
  • Comunicación: métodos Java internos → HTTP/REST
  • Estado: HttpSession en servidor → JWT en cliente
  • Vistas: JSP renderiza en servidor → React renderiza en navegador
  • API: implícita (Servlet forward) → explícita (Swagger docs)
  • Despliegue: 1 WAR → N contenedores Docker
  • Escalamiento: vertical (más RAM a Tomcat) → horizontal (más réplicas del servicio)

¿Por qué esta guía enseña ambos enfoques?

El monolito JSP te enseña los fundamentos: cómo funciona realmente el HTTP, las sesiones, el MVC, el JDBC. El stack moderno FastAPI + React te muestra la evolución natural: APIs REST, JWT, SPA, contenedores. No son enfoques rivales — el primero es la base conceptual del segundo. Los ingenieros que entienden el monolito construyen mejores microservicios porque saben qué problema resuelve cada abstracción.
